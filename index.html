collaboration

  • At NAP this started with a move to form cross-compentency teams
  • Creating small, focused product teams
  • Teams made of a product owner / engineers / testers and experience designers

empathy

  • For me this is one of the most crucial points
  • As it builds empathy between team members
  • And as that empathy Establishes itself it grows a culture of listening and learning within the team - self development if you will
  • Making our teams able to pivot and accelerate through change by making smarter decisions

challenging

  • These smaller teams also promote self-development through conversation
  • and that is a natural catalyst to challenging existing models
  • For our team we started to challenge the way we approached showcasing work to our stakeholders
  • Previously this process was to show full formed pages designs at the end of two week sprints, which tended resulted in
  • + subjective commentary + constant rework which was boring and promoted inconsistancy throughout
  • how many times do you want to design and develop a form field
  • ultimately began to derail our ability to deliver projects.

atomic

  • to move away from this we focused on taking a step back
  • and began showcasing elements at a much more granular level
  • using a methodolgy from the Brad Frost school of design
  • Atom - an element that can't be broken down anymore - H1
  • Molecules - a collection of atoms - language select
  • Organism - a collection of molecules - global header
  • Our showcase materials loo like this now - a visual stylesheet
  • An incredible useful way of approaching our design
  • it meant the team, and all stakeholders are invested to thinking abouts elements at an individual component level.

function checkForWebComponentsOnPage() {

  document.addEventListener('WebComponentsReady', function() {
    var HTMLImports = document.querySelectorAll('link[rel="import"]');
    var customElementIds = [];
    [].forEach.call(HTMLImports, function(HTMLImport) {
      if(HTMLImport.import) {
        var domModules = HTMLImport.import.getElementsByTagName('dom-module');
        [].forEach.call(domModules, function(module) {
          customElementIds.push(module.id);
        });
      }
    });

    customElementIds.forEach(function(elementName){
      var elements = document.getElementsByTagName(elementName);
      [].forEach.call(elements, function(element) {
        if(element.constructor === HTMLElement){ /* Do stuff */ }
      });
    });
  });
}
  • Small bit of code to check for broken elements on a page.
  • Get all imports
  • For each one, get all dom-modules and store the id
  • Essentially getting a list of all the defined elements on the page
  • Check the constructor. If it hasnt been upgraded by Polymer, it won't have rendered.
  • We send data to GA for each broken element
  • This does happen sometimes! But not often enough to panic
  • Still trying to work out the conditions under which this happens, and how to minimise it.
  • Also send data to GA whenever an element makes an Ajax call which fails
  • Result is easy-to-spot problems

2. CORS
(Cross-Origin Really Sucks)

  • All our assets are requested through a sub-domain, including HTML imports
  • We started seeing lots of CORS errors in the console
  • Lots of testing, inc. blaming our CDN provider
  • Days of investigation => Safari!!
  • No same-origin policy for HTML (or fonts), so no Origin header, cached without CORS headers

3. IE & "parser-challenged elements"

  • There are certain elements which IE removes any dissallowed children nodes
  • The two main ones are select and table:
